# 桌宠互动核心发布阻塞修正设计

## 背景

互动核心 1.2.0 发布审查发现四条确定性问题：延迟单击可在下一手势期间切换窗口模式，普通互动可覆盖生活动作，睡眠双击可吞掉唤醒，以及生命存档 IPC 接受超大额外字段。本设计在保持既有视觉、生命数值和 IPC 职责边界的前提下关闭这些发布阻塞。

## 范围

- 协调延迟单击与后续指针会话。
- 明确睡眠点击语义。
- 修正临时动作优先级。
- 加固生命快照 IPC 输入。
- 增加对应单元和时序回归测试。

Windows 实机缩放、多显示器拖动和安装包验收继续由 Task 8 完成。

## 手势时序

### 延迟单击

清醒状态的单击继续使用 301ms 窗口区分双击。单击计时器到期时若存在活跃指针会话，PetShell 将该单击标记为待提交，并保持窗口模式稳定。

第二个指针会话结束后按以下规则处理：

- 双击：取消待提交单击，只执行双击收益和庆祝动作。
- 点击：先提交首个单击，再为超过双击窗口的第二次点击创建独立延迟提交。
- 抚摸或无收益候选：先结束当前会话，再提交首个单击；抚摸动作最后写入，保留高优先级反馈。
- 拖动：等待拖动终端请求完成，先提交首个单击，再播放落地动作。
- 取消、丢失捕获或卸载：清除待提交单击。

待提交单击只保存布尔状态。实际提交时读取最新生命 store，避免闭包中的睡眠状态过期。

### 睡眠点击

睡眠状态下的首次有效点击立即调用 `wake(Date.now())` 并播放 `waking`，同时清除双击候选。睡眠桌宠不会播放庆祝动作，也不会因唤醒点击切换状态面板。

## 动作优先级

动作选择器按以下顺序返回首个可用动作：

1. 高优先级互动：`dragging`、`petting`、`dropping`。
2. 生活动作：`eating`、`falling-asleep`、`waking`。
3. 普通互动：`waving`、`celebrating`。
4. 业务动作：任务成功、任务失败、额度不足。
5. 当前生命形态。

该顺序允许拖动、抚摸和落地反馈即时响应，同时保证喂食、入睡和唤醒动画不会被普通点击覆盖。

## IPC 输入边界

生命快照仅接受以下六个字段：

- `mood`
- `satiety`
- `energy`
- `sleeping`
- `lastCalculatedAt`
- `lastInteractionAt`

preload 在调用 `ipcRenderer.invoke` 前依次执行 UTF-8 JSON 大小、字段集合和基础类型校验。Main 在信任边界再次执行相同结构校验，并继续由存储 normalizer 执行数值范围和时间规则校验。

IPC 快照上限采用 16 KiB，与磁盘读取上限一致。校验失败统一抛出固定生命状态保存错误；错误文本不包含 payload、路径或字段值。

## 错误处理

- 待提交单击在终端取消路径中直接丢弃。
- 拖动结束失败继续清除拖动视觉，且不会播放落地动作。
- preload 和 Main 的生命快照拒绝使用固定错误文案。
- 校验逻辑不得记录快照内容。

## 测试

- 使用 fake timer 覆盖首个单击到期时第二会话仍活跃、第二会话为双击、抚摸、拖动和取消的结果。
- 覆盖睡眠首次点击立即唤醒及睡眠双击不会庆祝。
- 穷尽测试高优先级互动、生活动作、普通互动、业务动作和生命形态的排列。
- preload 与 Main 分别测试额外字段、超过 16 KiB、错误类型和合法快照。
- 运行 `npm run verify`、生产依赖审计和 Windows workflow。

## 验收标准

- 第二个指针会话期间不会发生窗口模式切换。
- 睡眠桌宠一次点击即可唤醒。
- 普通点击和双击不会覆盖进行中的生活动作。
- 超大或包含额外字段的生命快照在 preload 和 Main 两侧均被拒绝。
- 全量测试、类型检查、bundle 边界、Vite 构建和生产依赖审计通过。
